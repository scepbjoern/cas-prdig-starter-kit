## Schritt 5.1.2 – Route Handler `app/api/antraege/route.ts` erstellen
**Ziel:** `GET /api/antraege` → Liste, `POST /api/antraege` → Erstellen.

**Datei:** `src/app/api/antraege/route.ts`

**Wichtig:** In Route Handlern **nicht** `requireSession()` verwenden! Diese Funktion ruft `redirect('/login')` auf (HTML-Redirect), was für eine JSON-API ungeeignet ist. Stattdessen `getSession()` aus `@/lib/auth-helpers` nutzen und bei fehlender Session `NextResponse.json({ error: 'Nicht authentifiziert' }, { status: 401 })` zurückgeben.

**GET-Logik:**
1. Session holen via `getSession()`. Ohne Session → 401.
2. Rolle prüfen: `user_applicant` sieht nur eigene Anträge; `admin`/`user_reviewer` sehen alle.
3. `prisma.antrag.findMany({ where: ..., include: { ersteller: { select: { name: true, email: true } } }, orderBy: { erstelltAm: 'desc' } })`
4. `NextResponse.json(antraege)` zurückgeben.

**POST-Logik:**
1. Session holen via `getSession()`. Ohne Session → 401.
2. Rolle prüfen: nur `user_applicant` oder `admin` dürfen erstellen. Andere → 403.
3. Body parsen: `const body = antragCreateSchema.parse(await req.json())`.
4. Bei `ZodError`: `NextResponse.json({ error: 'Ungültige Daten', details: error.errors }, { status: 400 })`.
5. Prisma: `prisma.antrag.create({ data: { ...body, erstellerId: session.user.id } })`.
6. `NextResponse.json(antrag, { status: 201 })` zurückgeben.

**Erwartetes Resultat:**
- `GET http://localhost:3000/api/antraege` → JSON-Liste mit Antrag-Objekten (mit Cookie).
- `POST http://localhost:3000/api/antraege` mit gültigem Body → erstelltes Objekt (201).
- `POST` mit ungültigem Body → 400 mit Fehlermeldung.
- `GET` ohne Cookie → 401 JSON.

## Schritt 5.1.3 – Route Handler `app/api/antraege/[id]/route.ts` erstellen
**Ziel:** `GET /api/antraege/:id` → Detail, `PUT /api/antraege/:id` → Aktualisieren, `DELETE /api/antraege/:id` → Löschen.

**Datei:** `src/app/api/antraege/[id]/route.ts`

**Wichtig – Next.js 16 `params` als Promise:** Seit Next.js 15 ist `context.params` ein Promise. Die Handler-Signatur muss lauten:
```ts
export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  // ...
}
```
Das gilt für ALLE Methoden (GET, PUT, DELETE) im `[id]/route.ts`.

**Auth wie in Schritt 5.1.2:** Auch hier `getSession()` statt `requireSession()` nutzen → JSON-Fehlerantworten (401/403) statt HTML-Redirects.

**GET-Logik:**
1. Session via `getSession()`. Ohne → 401.
2. `const { id } = await params`; via `antragIdSchema.safeParse(id)` validieren. Ungültig → 400.
3. `prisma.antrag.findUnique({ where: { id }, include: { ersteller: { select: { name: true } } } })`.
4. Nicht gefunden: `NextResponse.json({ error: 'Nicht gefunden' }, { status: 404 })`.
5. Applicant darf nur eigene Anträge sehen → sonst 403.
6. Gefunden: `NextResponse.json(antrag)`.

**PUT-Logik:**
1. Session + `params` validieren.
2. Rolle prüfen: `user_applicant` nur eigene + nur `ENTWURF`; `admin` alles; `user_reviewer` via PUT nicht erlaubt → 403.
3. Body via `antragUpdateSchema.safeParse(await req.json())` validieren.
4. Prisma: `prisma.antrag.update({ where: { id }, data: body })`.
5. `NextResponse.json(antrag)` zurückgeben.

**DELETE-Logik:**
1. Session + `params` validieren.
2. Rolle prüfen: Admin immer; Applicant nur eigene + nur `ENTWURF` → sonst 403.
3. Prisma: `prisma.antrag.delete({ where: { id } })`.
4. `NextResponse.json({ message: 'Gelöscht' })`.

**Fehlerbehandlung (alle Methoden):**
- `ZodError` → 400 + details.
- `PrismaClientKnownRequestError` (P2025) → 404.
- Unauthentifiziert → 401.
- Unberechtigt → 403.
- Sonstige → 500 mit generischer Nachricht.

**Erwartetes Resultat:**
- `GET /api/antraege/<id>` → 200 mit JSON oder 400/404.
- `PUT /api/antraege/<id>` → 200 mit aktualisiertem JSON oder 400/403/404.
- `DELETE /api/antraege/<id>` → 200 oder 400/403/404.

## Schritt 5.1.4 – Middleware: `/api/antraege` in PUBLIC_PATHS aufnehmen
**Ziel:** Middleware soll `/api/antraege` nicht per Redirect auf `/login` schützen, sondern die Auth-Prüfung den Route Handlern überlassen (die JSON-Fehlerantworten zurückgeben).

**Datei:** `middleware.ts` (anpassen)

**Änderung:** `PUBLIC_PATHS` um `/api/antraege` erweitern:
```ts
const PUBLIC_PATHS = ['/login', '/api/auth', '/api/antraege']
```

**Begründung:** Die Middleware leitet unauthentifizierte Requests auf `/login` um (HTML-Redirect). Für API-Clients ist ein Redirect auf eine Login-Seite unbrauchbar. Die Route Handler haben eigene Auth-Logik via `getSession()` und geben korrekte 401/403 JSON-Antworten. Die doppelte Auth-Prüfung (Middleware + Route Handler) würde nur den Redirect-Fall erzeugen, bevor der Route Handler seine JSON-Antwort senden kann.

**Alternative (falls PUBLIC_PATHS nicht erweitert werden soll):** Middleware belassen, akzeptieren dass API-Calls ohne Cookie einen Redirect bekommen. Dies ist für lokale Entwicklung mit Cookie-basierter Auth akzeptabel, aber nicht REST-konform.

**Empfehlung:** PUBLIC_PATHS erweitern.

## Schritt 5.1.5 – Authz-Regeln korrekt umsetzen
Die Route Handler müssen die gleichen Autorisierungsregeln wie die Server Actions (`src/app/(app)/antraege/actions.ts`) befolgen:

| Aktion | Erlaubt für |
|---|---|
| `GET /api/antraege` | Alle Rollen (applicant sieht nur eigene) |
| `POST /api/antraege` | `user_applicant`, `admin` |
| `GET /api/antraege/:id` | Alle Rollen (applicant nur eigene) |
| `PUT /api/antraege/:id` | Eigentümer (nur ENTWURF) + `admin` |
| `DELETE /api/antraege/:id` | Eigentümer (nur ENTWURF) + `admin` |
