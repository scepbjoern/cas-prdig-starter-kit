# Phase 4: UI-Aufbau (Layout + shadcn/ui + CRUD)

## Ziel
Vollständige, rollenbasierte UI mit Layout, Auth-geschützten Seiten und CRUD-Flows für die Demo-Entitäten `Antrag` und `Person`. Nach Abschluss: `npm run build` läuft fehlerfrei, alle Rollen können sich anmelden und ihre Bereiche bedienen.

## Architektur-Entscheidungen
- **Route Groups:** Geschützte Seiten unter `src/app/(app)/...` mit gemeinsamem Layout (Sidebar + Topbar). `/login` bleibt ausserhalb.
- **Forms:** Client-Forms mit `react-hook-form` + Zod (shadcn `<Form>`); Submit ruft Server Action auf, die identisches Zod-Schema serverseitig validiert.
- **Auth-Check:** Middleware bleibt für Cookie-Check zuständig; Server Components nutzen einen zentralen `requireSession()`-Helper, Server Actions zusätzlich `requireRole()`.
- **Status-Feedback:** `sonner` (shadcn Toast) für Erfolg/Fehler nach Server Actions.
- **Tests:** Bewusst NICHT Teil dieser Phase – kommen in Phase 7.

## Voraussetzungen / zusätzliche shadcn-Komponenten
Bereits vorhanden: `button`, `card`, `input`, `label`, `select`, `table`, `badge`, `avatar`, `sheet`, `separator`, `dropdown-menu`.

Zusätzlich zu installieren via `npx shadcn@latest add ...`:
- `form` – RHF-Integration (`Form`, `FormField`, `FormItem`, `FormLabel`, `FormControl`, `FormMessage`)
- `textarea` – für `Antrag.beschreibung`
- `dialog` – Bestätigungs-Dialoge (z.B. Löschen)
- `sonner` – Toaster für Action-Feedback
- `skeleton` – für `loading.tsx`-States

## Schritt 4.0 – Fundament (Helper, Konstanten, Provider)

### 4.0.1 Auth-Helper `src/lib/auth-helpers.ts`
- `getSession()` – wrappt `auth.api.getSession({ headers: await headers() })`
- `requireSession()` – wirft Redirect auf `/login`, wenn keine Session
- `requireRole(roles: Role[])` – nutzt `requireSession`, prüft `session.user.role`, wirft `forbidden()` (404) oder Redirect auf `/`
- Type-Export `Role = 'admin' | 'user_reviewer' | 'user_applicant'`

### 4.0.2 Status-Mapping `src/lib/antrag-status.ts`
- `ANTRAG_STATUS_LABEL: Record<AntragStatus, string>` (DE-Labels)
- `ANTRAG_STATUS_VARIANT: Record<AntragStatus, BadgeVariant>` (z.B. ENTWURF → secondary, EINGEREICHT → default, GENEHMIGT → success-Variante, ABGELEHNT → destructive)
- `ANTRAG_STATUS_TRANSITIONS: Record<AntragStatus, AntragStatus[]>` – erlaubte Wechsel pro Rolle (Antragsteller: ENTWURF→EINGEREICHT; Reviewer/Admin: EINGEREICHT→GENEHMIGT|ABGELEHNT)

### 4.0.3 Toaster-Provider
- `<Toaster />` aus `sonner` ins Root-Layout (`src/app/layout.tsx`) einfügen.

### 4.0.4 Navigation-Konfiguration `src/lib/navigation.ts`
- Zentrale Liste: `{ href, label, icon, roles: Role[] }[]`
- Wird von `Sidebar` konsumiert; Filterung pro Rolle erfolgt deklarativ statt verstreut.

**Akzeptanz:** TypeScript kompiliert, keine UI-Änderung sichtbar.

## Schritt 4.4 – Demo-Entität: Anträge

### 4.4.1 Schema & Actions
- `src/lib/schemas/antrag.ts` – `antragCreateSchema`, `antragUpdateSchema`, `antragStatusSchema`
- `src/app/(app)/antraege/actions.ts` – Server Actions:
  - `createAntrag(formData)` – Antragsteller darf erstellen (Status ENTWURF), revalidiert `/antraege`
  - `updateAntrag(id, formData)` – Ersteller darf eigene ENTWURF-Anträge editieren
  - `submitAntrag(id)` – ENTWURF → EINGEREICHT (nur Ersteller)
  - `decideAntrag(id, status)` – EINGEREICHT → GENEHMIGT|ABGELEHNT (nur reviewer/admin), nutzt `ANTRAG_STATUS_TRANSITIONS`
  - `deleteAntrag(id)` – nur Admin oder Ersteller eines ENTWURFs
  - Alle Actions: `requireSession()`, Zod-Parse, Authz-Check, `revalidatePath`

### 4.4.2 Listenseite `src/app/(app)/antraege/page.tsx` (Server Component)
- Rollenbasierte Query:
  - `user_applicant`: `where: { erstellerId: session.user.id }`
  - `user_reviewer`: `where: { status: 'EINGEREICHT' }` (default-View)
  - `admin`: alle
- shadcn `Table` mit Spalten: Titel, Status (`Badge` mit Variant aus Mapping), Ersteller, Erstellt am, Aktionen
- Header-Action: `<Link href="/antraege/neu">Neuen Antrag</Link>` (nur applicant + admin)
- Empty-State (Card mit Hinweis)

### 4.4.3 Formular-Komponente `src/components/antraege/antrag-form.tsx`
- Wiederverwendbar für Create + Edit
- RHF + Zod, shadcn `Form` Komponenten, `Textarea` für Beschreibung
- Submit-Handler ruft entsprechende Server Action auf

### 4.4.4 Neu `src/app/(app)/antraege/neu/page.tsx`
- Server Component prüft Rolle (`requireRole(['user_applicant', 'admin'])`)
- Rendert `<AntragForm mode="create" />`

### 4.4.5 Detail `src/app/(app)/antraege/[id]/page.tsx`
- Server Component lädt Antrag inkl. Ersteller; `notFound()` wenn nicht existent
- Authz: Ersteller, Reviewer, Admin
- Drei Bereiche:
  - **Stammdaten** (Titel, Beschreibung, Erstellt am, Ersteller)
  - **Status-Block** mit Badge + erlaubten Aktionen je Rolle:
    - ENTWURF + Ersteller: Bearbeiten-Link, Einreichen-Button
    - EINGEREICHT + Reviewer/Admin: Genehmigen / Ablehnen Buttons (Confirm-`Dialog`)
  - **Bearbeiten-Modus** (nur ENTWURF + Ersteller): inline AntragForm `mode="edit"`
- Action-Buttons sind kleine Client-Components mit `useTransition` + Toast-Feedback

### 4.4.6 `loading.tsx` & `error.tsx` für `/antraege` und `/antraege/[id]`

**Akzeptanz:** Antragsteller kann erstellen, bearbeiten (nur Entwürfe), einreichen. Reviewer kann genehmigen/ablehnen. Admin kann alles. Status-Badges + Toasts erscheinen korrekt.

## Offene Punkte / Tradeoffs

1. **`requireRole` vs. UI-Filter:** Sicherheit liegt auf Server-Action-Ebene (Pflicht). UI-Filter in Sidebar/Buttons ist nur UX, keine Security-Boundary.
2. **Pagination:** Bei Demo-Datenmenge nicht nötig, aber als Kommentar im Code erwähnen.
3. **Optimistic Updates:** Für Status-Änderungen sinnvoll, aber Toast + `revalidatePath` reichen für ein Starter-Kit.
4. **i18n:** Plan ist DE-only; keine Vorbereitung für i18n-Library.
